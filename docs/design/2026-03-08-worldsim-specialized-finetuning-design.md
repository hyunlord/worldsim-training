# WorldSim 특화형 파인튜닝 설계 — "좁고 깊게, 변형은 데이터로"

> **핵심 원칙**: 0.8B에게 "자유 창작"을 시키지 않는다.
> "정해진 슬롯을 채우는 법"만 가르치되, 슬롯 내용의 변형을 데이터로 미리 심어둔다.

---

## 1. 설계 철학: 자유도를 코드가 통제하고, LLM은 "칸 채우기"만 한다

### 기존 접근 (실패한 것)
```
프롬프트: "이 에이전트의 성격과 감정을 바탕으로 자유롭게 묘사해라"
→ 모델이 할 일: 무한한 가능성에서 적절한 문장을 "창작"
→ 결과: "이 사람은 이다. 이 사람은 이다. 이 사람은 이다." (degeneration)
```

### 새 접근 (특화형)
```
프롬프트: "[겁많음][두려움:강함][짐승발견] → 2문장 묘사"
→ 모델이 할 일: 학습한 (성격, 감정, 상황) 조합 패턴에서 적절한 문장 "선택+조합"
→ 결과: "풀숲 너머로 날랜 짐승의 그림자를 보았다. 온몸이 오들오들 떨렸다."
```

**차이점**: 모델이 "창작"하는 게 아니라, 학습 데이터에서 본 비슷한 패턴을 "재조합"하는 것. 0.8B가 잘할 수 있는 범위.

## 2. Task 정의: 정확히 4가지만 학습시킨다

### Task A: 성격 한줄 묘사 (1문장)
```
입력: [H:0.8][E:0.2][X:0.7][A:0.4][C:0.9][O:0.3]
출력: "곧은 마음에 겁 없고, 무리 앞에 잘 나서나, 한번 마음먹으면 끝을 보는 이다."
```
- **고정**: 항상 1문장, 20-40자
- **변동**: 성격 조합에 따라 어휘가 달라짐
- **학습 포인트**: HEXACO 6축 → 한국어 성격 형용사 매핑

### Task B: 상황 반응 묘사 (2문장)
```
입력: [겁많음][두려움:강함][상황:짐승발견]
출력: "풀숲 너머로 날랜 짐승의 그림자를 보았다. 온몸이 오들오들 떨리며 뒷걸음질 쳤다."
```
- **고정**: 항상 2문장, 30-60자
- **변동**: (성격 × 감정 × 상황) 조합
- **학습 포인트**: 의성어/의태어 사용, 행동 묘사

### Task C: 짧은 대사 (1문장 직접화법)
```
입력: [우두머리][분노:중간][상황:도둑적발][어투:하오체]
출력: "누가 먹거리를 훔쳤소! 당장 나서시오!"
```
- **고정**: 항상 1문장 대사, 15-30자, 지정된 어투
- **변동**: 역할+감정에 따라 어휘/어투 변화
- **학습 포인트**: 해라체/하오체/해체 종결어미 + 역할별 말투

### Task D: 알림 카피 (1문장)
```
입력: [이름:돌이][상황:먹거리발견]
출력: "돌이가 강가에서 물고기를 잡았다."
```
- **고정**: 항상 1문장, 10-25자, 해라체 고정
- **변동**: 이름 + 상황에 따라 내용 변화
- **학습 포인트**: 간결한 사실 전달

## 3. 입출력 형식: 엄격히 고정

### 입력 형식 (모든 Task 공통 구조)

```
[TASK] B
[NAME] 돌이
[PERS] 겁많음, 꼼꼼함, 조용함
[EMOT] 두려움:0.8
[SITU] 짐승발견
[REG] 해라체
[LEN] 2문장
```

- 태그 기반 구조화 입력 — JSON이 아닌 태그로 (토큰 절약)
- 모든 필드가 짧은 키워드 — 긴 설명문 없음
- 입력 토큰: **50-80 토큰** (컨텍스트 윈도우 절약)

### 출력 형식 (Task별 고정)

| Task | 출력 형식 | 길이 제한 | 어투 |
|:----:|----------|:--------:|:----:|
| A | 평서문 1문장 | 20-40자 | 해라체 |
| B | 묘사문 2문장 | 30-60자 | 해라체 |
| C | 대사 1문장 (큰따옴표) | 15-30자 | 지정됨 |
| D | 알림 1문장 | 10-25자 | 해라체 |

**출력에 JSON 없음.** 순수 텍스트만. GBNF로 길이/형식 제약.

## 4. 학습 데이터 설계

### 규모 (기존 16K-23K에서 축소)

| Task | 조합 수 | 변형/조합 | 총 예시 | 비고 |
|:----:|:-------:|:--------:|:------:|------|
| A (성격 한줄) | 128 아키타입 | 5 변형 | **640** | LHSMDU 128개 × 5문장 변형 |
| B (상황 반응) | 10 상황 × 8 감정 × 10 성격군 | 3 변형 | **2,400** | 핵심 조합만 |
| C (대사) | 10 상황 × 3 어투 × 10 성격군 | 3 변형 | **900** | 어투 학습이 핵심 |
| D (알림) | 20 상황 | 5 변형 | **100** | 가장 단순 |
| **합계** | | | **~4,000** | |
| 대조 예시 (부정) | | | **~1,000** | 잘못된 출력 예시 |
| 일반 한국어 (forgetting 방지) | | | **~500** | |
| **총 합계** | | | **~5,500** | |

**기존 16K-23K에서 5.5K로 축소.** 이유:
- Task가 4개로 좁아짐 → 적은 데이터로 커버 가능
- "빈칸 채우기"는 "자유 창작"보다 훨씬 적은 예시로 학습 가능
- LIMA 원칙: 5K 고품질 > 20K 저품질

### 데이터 생성 전략

**Teacher 모델**: Claude Sonnet API (전량 — $5.5K × $3.90/1K = ~$22)

기존 계획(90% 로컬 35B + 10% API)을 **100% API**로 변경. 이유:
- 5.5K는 소량 → API 비용이 $22 수준으로 충분히 저렴
- Teacher 모델 품질이 학습 데이터 품질에 직결 → Sonnet이 35B보다 확실히 좋음
- "좁은 범위"이므로 품질 일관성이 더 중요

### 대조 예시 (부정 데이터) 1,000개

| 유형 | 예시 | 학습 목적 |
|------|------|----------|
| 반복 루프 | "이 사람은 이다. 이 사람은 이다." | 이런 출력을 하지 말 것 |
| 한자어 사용 | "식량이 부족하여 전투를 준비했다" | 한자어를 쓰지 말 것 |
| 어투 혼용 | "위험하오... 빨리 뛰어!" (하오+해체 혼합) | 어투를 섞지 말 것 |
| 과도한 길이 | 5문장 이상의 장황한 출력 | 지정 길이를 초과하지 말 것 |
| 현대어 | "OK, 알겠어. 바로 가자" | 현대어를 쓰지 말 것 |

이걸 **DPO(Direct Preference Optimization)** 또는 단순히 "이런 출력은 나쁨" 라벨로 학습.

## 5. 유동성이 생기는 메커니즘

### 5.1 데이터 내 변형 (학습 시점)

같은 조건에 대해 3-5개 변형을 넣어둠:

```
입력: [겁많음][두려움:강함][짐승발견]

변형1: "풀숲 너머로 날랜 짐승의 그림자를 보았다. 온몸이 오들오들 떨렸다."
변형2: "어둠 속에서 으르렁거리는 소리가 들렸다. 다리가 풀려 주저앉았다."
변형3: "날카로운 이빨이 달빛에 번들거렸다. 숨이 턱 막혀 꼼짝도 못 했다."
```

모델은 이 변형들의 **패턴**(짐승 묘사 → 신체 반응)을 학습하고, 추론 시 temperature 0.7에서 자연스럽게 변형 사이를 오감.

### 5.2 입력 변수 조합 (추론 시점, Rust 코드)

Rust 코드가 **입력을 조합**해서 다양성을 만듦:

```rust
// 같은 "짐승발견" 상황이라도 에이전트마다 다른 입력이 들어감
agent_1: [겁많음][두려움:0.9][짐승발견]  → 극도의 공포 반응
agent_2: [대담함][분노:0.6][짐승발견]    → 공격적 반응  
agent_3: [호기심][놀람:0.7][짐승발견]    → 관찰하는 반응
```

LLM은 같은 상황에서도 **입력 성격+감정에 따라 다른 출력**을 내놓게 학습됨. 유동성은 LLM의 "창의성"이 아니라 **입력 조합의 다양성**에서 나옴.

### 5.3 의성어/의태어 풀 (데이터에 내장)

학습 데이터에 감정별 의성어를 다양하게 심어둠:

| 감정 | 의성어/의태어 풀 |
|------|----------------|
| 두려움 | 오들오들, 덜컥덜컥, 벌벌, 후들후들 |
| 기쁨 | 방긋, 덩실덩실, 싱글벙글 |
| 분노 | 부들부들, 벌컥, 으드득 |
| 슬픔 | 그렁그렁, 글썽글썽, 훌쩍 |

모델이 이 풀에서 골라 쓰도록 학습. 새로운 의성어를 "발명"하는 건 아님.

## 6. GBNF Grammar (추론 시 출력 강제)

### Task A/B/D용 (해라체 평서문)
```gbnf
root     ::= sentence (". " sentence)? "."?
sentence ::= word (ws word){3,20}
word     ::= [가-힣]+
ws       ::= " "
```

### Task C용 (대사, 큰따옴표)
```gbnf
root     ::= "\"" sentence "!\""  |  "\"" sentence ".\""  |  "\"" sentence "?\""
sentence ::= word (ws word){2,15}
word     ::= [가-힣]+
ws       ::= " "
```

핵심: **한글만 허용, 영어/숫자/JSON 구조 불허.** 출력 길이를 단어 수로 제한.

## 7. 실행 계획

### Phase A: 데이터 생성 (1-2일)

1. 10개 상황 정의 (짐승발견, 먹거리발견, 비바람, 부상, 친족사망, 도둑적발, 무리이동, 도구제작, 불발견, 낯선무리)
2. 10개 성격군 정의 (Gemini 10개 아키타입 기반)
3. 8개 감정 (Plutchik 기본)
4. Sonnet API로 5,500개 생성 (~$22, 수 시간)
5. 자동 QA: 길이 체크 + 한자어 체크 + 어투 체크

### Phase B: QLoRA 학습 (수 시간)

```
모델: Qwen3.5 0.8B
방법: QLoRA r=16, alpha=32
데이터: 5,500예시 (4,000 정상 + 1,000 대조 + 500 일반)
Epoch: 5 (소량 데이터이므로 더 돌림)
커리큘럼: Task D(쉬움) → Task A → Task B → Task C(어려움)
시간: DGX Spark에서 ~30분-1시간
```

### Phase C: 평가 (반나절)

- 50개 holdout 예시로 자동 평가
- 반복 발생률 < 5%
- 길이 준수율 > 90%
- 한자어 비율 < 10%
- 어투 일치율 > 85%
- **사람 평가**: 본인이 10개 읽고 "자연스러운가?" 확인

### Phase D: GGUF 변환 + 게임 적용 (수 시간)

- LoRA merge → GGUF Q4_K_M
- 기존 모델 교체
- 게임 실행 확인

**총 소요: 3-4일, ~$22**

## 8. 게임 코드 변경

### Rust 프롬프트 빌더 수정

기존의 긴 Jinja 템플릿 대신 태그 기반 단순 포맷:

```rust
fn build_task_b_prompt(
    personality_keywords: &[&str],  // ["겁많음", "꼼꼼함"]
    emotion: &str,                   // "두려움"
    emotion_intensity: f64,          // 0.8
    situation: &str,                 // "짐승발견"
) -> String {
    format!(
        "[TASK] B\n[PERS] {}\n[EMOT] {}:{:.1}\n[SITU] {}\n[REG] 해라체\n[LEN] 2문장",
        personality_keywords.join(", "),
        emotion,
        emotion_intensity,
        situation,
    )
}
```

### 시스템 프롬프트 (파인튜닝 모델용, 짧게)

```
너는 석기시대 서사 도우미다. 지시된 형식과 길이를 정확히 지켜라.
순우리말만 써라. 한자어 금지. 지정된 어투를 지켜라.
```

파인튜닝된 모델은 태그를 이미 "알고 있으므로" 긴 설명이 불필요.

## 9. "이게 진짜 충분한가?" 체크리스트

| 질문 | 답 |
|------|---|
| 0.8B가 1-2문장 한국어를 쓸 수 있나? | ✅ 가능 (지금도 짧은 문장은 OK, 길어지면 붕괴) |
| 태그 입력 → 특정 출력 매핑을 학습할 수 있나? | ✅ 이건 사실상 분류+슬롯 채우기 (0.8B의 강점) |
| 5,500개면 충분한가? | ✅ LIMA: 1K도 충분. Task가 4개뿐이라 5.5K은 넉넉 |
| 반복 루프가 사라질까? | ⚠️ 대폭 감소하지만 0은 아님 → GBNF 길이 제한 + validator가 잡음 |
| 성격 차이가 느껴질까? | ✅ PsychAdapter: 0.8B급에서 74% 구분 가능 |
| 순우리말을 지킬까? | ⚠️ 70-80% → validator가 나머지 자동 치환 |
| 파인튜닝 실패하면? | fallback 템플릿이 항상 있음. 최악의 경우에도 게임은 동작 |
